# SVG Studio — Design

> Implements STUDIO_PLAN.md Feature B (Option 3: verse-aware orchestrator).
> The 2026-07-08 decisions are taken as settled and are not revisited here:
> embedded **pywebview** preview with system-browser fallback; generator-owned
> files **detected and routed, never blocked**; **Inkscape optional** with a
> guided install prompt. This document is design only — no code exists yet.

## A correction to the plan's vocabulary (important)

STUDIO_PLAN.md treats Acts 13–14 as the "pure hand-authored one-off" example.
Reading the code: **every shipped SVG today is generator-owned**, including
Acts 13–14 — `generate_acts1314_svg.py` hand-*encodes* the geometry in Python
constants (`CITIES`, `ROUTE`, `LEGS`, `MAINLAND`…) but re-running it rewrites
all 80 files. So:

- The registry (§4) must cover Acts 13–14 like every other family.
- "Hand-authored one-offs" — files with **no** generator — do not exist yet.
  They are the *future* files this studio will let you create directly. The
  "start from previous verse's final frame" tool (§3) exists for exactly that
  future workflow, and for one-off overrides created via "hand-edit anyway".

## 2. The preview

### One URL, two hosts

Everything renders from a local HTTP server, so the embedded window and the
system-browser fallback are the *same* page — the fallback is "open the same
URL in the default browser", nothing else changes.

`PreviewServer` (daemon thread in the studio process, `127.0.0.1`, port 0 →
read the bound port) serves:

| Route | Serves |
|---|---|
| `/preview.html` | The wrapper page (template below). |
| `/verse.svg` | The **working buffer** — the XML pane's current text, not the saved file. `Cache-Control: no-store`. |
| `/verse.mp3` | The verse's narration MP3 if one exists (resolution in §2.4), else 404. |
| `/state` | `{"gen": <int>, "audio": <bool>, "ref": "Numbers 34:6 (KJV)"}` — `gen` is a generation counter the studio bumps on every (debounced) pane edit and on verse navigation. |

Threading: tkinter's mainloop and the HTTP thread never touch each other's
objects directly. The pane's `<<Modified>>` handler (debounced 400 ms via
`after`) copies the pane text into a lock-guarded `(gen, bytes)` pair the
server thread reads. That is the entire cross-thread surface.

### 2.2 The wrapper HTML

Dark page (`#101018`, matching the stage), SVG injected **inline** — not via
`<img>`/`<object>` — because re-inserting inline SVG into the DOM is the one
reliable way to restart SMIL from t=0, and it lets Replay sync audio:

```html
<body>
  <div id="stage"></div>
  <audio id="narration" src="/verse.mp3"></audio>
  <button id="replay">Replay ▶</button> <span id="ref"></span>
  <script>
    let gen = -1;
    async function tick() {
      const s = await (await fetch('/state')).json();
      if (s.gen !== gen) { gen = s.gen; await reload(s); }
    }
    async function reload(s) {
      const svg = await (await fetch('/verse.svg?g=' + gen)).text();
      document.getElementById('stage').innerHTML = svg;   // restarts SMIL
      const a = document.getElementById('narration');
      if (s.audio) { a.currentTime = 0; a.play().catch(()=>{}); }
    }
    document.getElementById('replay').onclick = () => reload(lastState);
    setInterval(tick, 500);
  </script>
```

(Real implementation keeps `lastState`, sets `#ref`, hides Replay when no
audio; this sketch pins the mechanism.) Live reload is therefore: type in the
XML pane → 400 ms debounce bumps `gen` → poller sees it within 500 ms →
re-inject → animation replays with narration. Sub-second, no websockets, no
dependencies.

Malformed XML mid-edit: the page injects whatever it gets; the browser shows
a broken/partial render, which is honest live feedback. The studio
additionally runs `xml.etree.fromstring` on the debounced buffer and shows a
parse error in the status line without blocking the preview.

### 2.3 The pywebview host — process model

`webview.start()` **blocks** and wants to own its thread/loop; on Windows it
spins a .NET/WinForms message pump via pythonnet. Running that inside the
tkinter process invites the exact event-loop fight §7 warns about. So the
embedded preview is a **subprocess**:

```
studio (tkinter + PreviewServer thread)
   └── subprocess: python svg_preview.py --url http://127.0.0.1:PORT/preview.html
         └── webview.create_window(title, url, width=1024+40, height=576+120)
             webview.start()          # blocks; process exits when window closes
```

- Env for the subprocess includes
  `WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS=--autoplay-policy=no-user-gesture-required`
  so narration can autoplay on reload without a click (WebView2 honors this;
  the system-browser fallback does *not*, which is why the Replay button
  exists and why `a.play()` failures are swallowed).
- The studio holds the `Popen` handle. "Open preview" reuses a live subprocess
  (it's already polling; nothing to do) or spawns one. Closing the studio
  terminates it.
- No IPC needed beyond the URL argument — all state flows through HTTP.

### 2.4 Narration MP3 resolution

Mirror the audio naming `audio_studio`/`generate_showcase_audio` write:
probe `web/audio/{Book}_{c}_{v}.{CODE}.mp3` for the studio's selected
translation variant, then `web/audio/{Book}_{c}_{v}.mp3`. First hit is served
at `/verse.mp3`. (Filesystem probe, not the audio manifest — the studio is on
the authoring machine and the probe matches the suffix-chain idiom used
everywhere else.)

### 2.5 Fallback: exact conditions

Fall back to `webbrowser.open(url)` — same page, live reload and Replay all
work — when **any** of:

1. `import webview` raises in a first-launch probe (run once:
   `python -c "import webview"` as a subprocess with a 15 s timeout).
2. The preview subprocess exits with a nonzero code, or exits at all, within
   10 s of spawn (covers: WebView2 runtime missing, pythonnet/clr-loader
   failing to find a .NET runtime, ARM64 wheel gaps — §7.1).
3. The preview subprocess dies twice within one studio session (covers
   crash-on-navigate class bugs).

On fallback the studio writes `{"preview_host": "browser"}` into its settings
(reuse the app's existing config mechanism) so subsequent launches skip the
probe and go straight to the browser, shows a one-line status explaining why,
and offers a "Try embedded preview again" toggle in the studio so a later
`pip install pywebview` fix can be picked up without editing config by hand.

pywebview itself is added to `requirements` as an **optional** extra, not a
hard dependency — the studio must import it lazily and only in the subprocess.

## 3. "Start from previous verse's final frame"

### Scope — where it applies and pointedly does not

- **Applies:** hand-authored sequences (future files with no generator) and
  one-off "hand-edit anyway" descendants. Button lives in the actions column.
- **Does NOT apply:** generator families. They already build cumulatively
  from the spec — `SvgAnimLayer.grow_rect(x_from=…)` holds the built part and
  grows only the remainder; `draw_wide` draws `verse > v` segments solid and
  `verse == v` traced. Freezing a generated file and editing on top would
  fork it from its spec. When the registry says the verse is generator-owned
  the button is **disabled** with tooltip: *"This family is generated; the
  generator already carries state between verses. Edit the spec instead."*

### Inputs and outputs

`freeze_to_base(svg_text: str, from_key: str) -> FreezeResult`
- `svg_text` — verse N−1's file content.
- `from_key` — e.g. `"Acts_13_4"`, recorded for lineage.
- `FreezeResult.svg_text` — the new starting document for verse N.
- `FreezeResult.warnings` — list of human-readable strings (shown in status /
  a dialog); never raises on the warning cases below.

### Algorithm (normative — no decisions left to the implementer)

Parse with `xml.etree.ElementTree` (namespace-aware; register the SVG default
namespace so serialization does not prefix every tag).

**Step 1 — classify every animation element.** Walk the whole tree for local
names `animate`, `set`, `animateTransform`, `animateMotion`.

For each node, in document order:

| Case | Action |
|---|---|
| `repeatCount="indefinite"` (any element type) | **Keep unchanged.** Perpetual animations (the tabernacle pulse) are steady-state ambience and part of the "final frame". |
| `<animate>` or `<set>` with `fill="freeze"` | **Freeze** (step 2), then remove the node. |
| `<animate>`/`<set>` with no `fill="freeze"` and finite repeat | The element visually snaps back to its base value at the end; the final frame *is* the base value. **Remove the node, change nothing else.** Warn: `"<animate> without fill=freeze removed (reverts to base value): <attributeName> on <parent-tag>"`. |
| `additive="sum"` or a `by=`-only animation | **Keep unchanged** and warn: freezing additive animation requires evaluating the base value, which may itself be animated; out of scope. (No shipped generator emits these.) |
| `animateTransform` / `animateMotion` (non-indefinite) | **Keep unchanged** and warn: `"animateTransform/animateMotion not frozen — element left animated"`. (No shipped generator emits these; deciding transform-composition order is not worth speculative code.) |

**Step 2 — the freeze itself**, for a qualifying `<animate>`/`<set>` on
parent element `P` with `attributeName="A"`:

1. Final value `V` = last `;`-separated entry of `values`, else the `to`
   attribute, else warn-and-keep (malformed).
2. Set `A="V"` as a presentation attribute on `P`, overwriting any existing
   `A`. (All house animations target presentation attributes:
   `stroke-dashoffset`, `fill-opacity`, `width`, `opacity`.) If two animate
   nodes target the same `A` on the same `P`, document order applies and the
   later one wins — matching SMIL's sandwich-model outcome for our
   non-additive case.
3. **Cleanup special case:** if `A == "stroke-dashoffset"` and `V == "0"`
   (numerically), instead of writing the attribute, *delete* both
   `stroke-dashoffset` and `stroke-dasharray` from `P`. A fully-revealed
   traced line becomes a plain polyline — visually identical, and the next
   author isn't confused by leftover dash plumbing.
4. `begin`, `dur`, `keyTimes`, `keySplines`, `calcMode` are ignored — the end
   state is the end state regardless of timing.

**Step 3 — demote to a flat base group.** Children of `<svg>` split into:
`<defs>` elements (keep in place, first) and everything else.

- Create `<g class="frozen-base" data-from="...">`. `data-from` = the new
  `from_key` **appended** to the previous base's `data-from` (space-
  separated), so lineage reads `"Acts_13_3 Acts_13_4"` after two hops.
- If the first non-defs child is already `<g class="frozen-base">` (verse N−1
  was itself produced by this tool), **hoist its children** into the new group
  instead of nesting — the base stays one flat group no matter how long the
  chain.
- Move all other non-defs top-level children into the group, in order.
- Append an empty `<g class="verse-overlay" id="v{N}">` after the base group
  with a comment child `<!-- verse N: add this verse's elements here -->`.
  New-element `id`s the author adds should be prefixed `v{N}-` (stated in the
  comment) to avoid collisions with frozen clip ids (`clip1`, …), which are
  left untouched.

`viewBox`/`width`/`height`/`font-family` on `<svg>` pass through unchanged.

**UI flow.** With verse N selected and no file existing for the chosen
variant, "Start from previous verse's final frame" loads verse N−1's file for
the *same variant chain* (`.CODE.svg` → `.svg`), runs `freeze_to_base`, puts
the result in the XML pane (unsaved), and shows warnings. It never writes to
disk itself — Save does.

**Testability.** `svg_freeze.py` gets unit tests with synthetic fixtures for
each table row above, plus round-trips of real house output: a `traced()`
polyline (dash attrs deleted), a `grown()` polygon (`fill-opacity` set to the
`hi` value), a `grow_rect` (`width` frozen to full), a pulse group (kept),
and the flat-base hoist across two consecutive freezes.

### UI flow for a generator-owned verse

Opening such a verse shows the banner (§1 layout) — the file loads into the
XML pane **read-only** until a choice is made:

- **"Edit spec (durable)"** (primary, default button):
  `os.startfile(spec.file)` for each spec pointer (opens in the user's
  editor — the studio does not embed a Python editor), and the banner swaps
  to a **[Re-run generator]** button. Re-run executes
  `[sys.executable, generator]` with `cwd`=repo root in a worker thread,
  streaming stdout to the status line; on exit 0 the studio reloads the
  current file from disk and bumps the preview generation. Note in the banner
  when `translation_suffixed` is true: re-run fetches passages for name
  resolution (network / bible cache).
- **"Hand-edit this SVG anyway (one-off)"** (secondary): unlocks the XML
  pane. On Save, the studio inserts (or updates) a marker comment as the
  first child of `<svg>`:
  `<!-- hand-edited YYYY-MM-DD; regenerating via generate_x.py will discard this -->`
  Purely informational — nothing enforces it, per the "never block" decision.
  If a file opened later already carries the marker, the banner mentions it
  ("this file has one-off hand edits") so a re-run isn't triggered casually.

Non-generator verses (registry miss) open straight into an editable pane, no
banner — and for a verse with no file at all, the pane offers a minimal
skeleton (`<svg viewBox="0 0 1024 576" …>` matching the house `W, H`) or the
§3 continue-from-previous flow.

## 5. The verse navigator

Book/chapter combos exactly as `audio_studio.py` builds them (from
`books.BOOK_NAMES` / `books.chapters_in`); verse spinner bounded by the
chapter's verse count when the passage is cached, else free-typed integer.

For the selected verse, scan `visuals/<Book>/<chapter>/` for
`{Book}_{c}_{v}*.svg` and decompose stems into variant chips:

```
variants:  [generic ●]  [KJV ●]  [WEB ○]  [ASV ○] …   (also: .webp exists)
```

- One chip per translation code from `passages.TRANSLATIONS` plus "generic".
  Filled dot = file exists; hollow = would fall back. Clicking a chip selects
  which file the pane edits; clicking a hollow chip offers "create as this
  variant" (skeleton, copy-from-generic, or §3 continue-from-previous).
- A resolution readout under the chips answers "what will each reader
  actually get", mirroring the shared suffix chain
  (`visuals.js findImage` / `visual_stage._find_image`):
  `KJV → Matthew_1_2.KJV.svg · WEB → Matthew_1_2.svg (generic) · ASV → generic`.
  Implementation is the same two-loop probe: for each translation, try
  `.CODE.svg` then `.svg` at the verse key only (chapter/book/default
  fallbacks are out of scope for an authoring tool — note it in a tooltip).
- If non-SVG assets exist for the key (`.webp` etc.), a muted note appears:
  the web app prefers SVG (`EXT_PRIORITY`), the desktop reader will keep
  using the raster. This is informational only.

The navigator also drives `/verse.mp3` resolution (§2.4) and re-checks the
registry on every book/chapter change.

## 6. Save / publish glue

- **Save** (`Ctrl+S`): validate XML (`fromstring`; on failure, dialog with
  the parse error and an "save anyway" escape hatch), write the pane bytes to
  the variant's path via the same `visuals/<Book>/<chapter>/` layout
  (`generate_tabernacle.out_path` is importable and already does this), add
  the hand-edit marker when §4 requires it.
- **Save + manifest**: after writing, call
  `build_manifest.build_manifest()` in-process (guard: `os.chdir` is global —
  instead run it as `subprocess([sys.executable, "build_manifest.py"],
  cwd=REPO_ROOT)` to avoid chdir races with the reader process's relative
  paths; it takes well under a second). Status line reports the key count.
  Plain Save skips the manifest (the manifest only feeds the web app; while
  iterating there is no reason to rebuild per keystroke). Default button is
  Save + manifest.
- **Stage for deploy** (optional, explicit): run
  `powershell -ExecutionPolicy Bypass -File prepare-deploy.ps1` with
  `cwd`=repo root in a worker thread, stream output to the status line. This
  mirrors the Audio Renderer's deploy-file-count discipline: before running,
  show `count_deploy_files()` (already implemented in `audio_studio.py`;
  import it) so the ~20k Cloudflare ceiling stays visible.
- The desktop reader needs no notification — it cannot render SVG; nothing to
  refresh. The web dev loop is: Save + manifest → whatever local static
  server the user runs over the repo sees the new file immediately.

## 7. Risks

### 7.1 pywebview on Windows ARM64 — the real one

pywebview's Windows backend is WebView2 (Edge Chromium), driven through
WinForms via **pythonnet**. Three links in that chain, assessed:

- **WebView2 runtime**: ships ARM64-native with Windows 11; not a risk on
  this machine.
- **pywebview itself**: pure Python; added explicit Edge Chromium ARM64
  support in 2022 and is actively maintained (6.x as of late 2025). Not a
  risk per se.
- **pythonnet + clr-loader + cffi**: this is the cairosvg-shaped hazard.
  pythonnet 3.x is a py3-none-any wheel but depends on `cffi` (compiled) and
  on locating a .NET runtime via `clr-loader`. `cffi` publishes win-arm64
  wheels only for recent Python versions, and clr-loader's runtime discovery
  on ARM64 has historically been the failure point. **This must be treated as
  "may simply not import" until proven on the machine.**

Mitigation is already designed in, not bolted on: the §2.5 probe (import
subprocess with timeout → 10-s liveness check → two-strikes rule) is the
named fallback trigger, the browser fallback is feature-complete (same URL,
same live reload), and the studio persists the outcome so a broken pywebview
costs one probe, once. Build order (§8) delivers the browser path *first*, so
the embedded window is an enhancement layer that can fail closed.

### 7.2 tkinter + webview event loops

Solved structurally: `webview.start()` never runs in the studio process
(§2.3). The only concurrency inside the studio process is the HTTP daemon
thread reading a lock-guarded buffer, and worker threads for generator
re-runs / deploy staging that report back via the existing
`queue + root.after` idiom `audio_studio.py` already uses. No tkinter object
is ever touched off the main thread.

Residual: orphaned preview subprocesses if the studio is killed hard.
Mitigation: the preview subprocess polls `/state`; when the server stops
answering (studio gone) for >10 s it closes itself.

### 7.3 SMIL freeze edge cases

Enumerated and given normative outcomes in §3's table (indefinite kept;
non-freeze removed with warning; additive/`by` kept with warning;
animateTransform/Motion kept with warning; dash-cleanup special case;
same-attribute ordering; flat-base hoisting). The deliberate posture:
**freeze what the house generators emit, warn-and-preserve everything else**
— a warned-but-still-animated element is visibly wrong in the preview, which
is exactly where you want to discover it.

### 7.4 Smaller ones

- **Autoplay policy**: embedded window gets the WebView2 browser-args
  override; system browser will block audio autoplay until first user gesture
  — Replay button is the designed gesture. Not a defect, but worth a line in
  the status bar in fallback mode.
- **SMIL in Chromium**: deprecated-but-supported for years and used by the
  shipped web app already; the preview inherits whatever the web app's fate
  is, so no *additional* risk.
- **Port/firewall**: bind `127.0.0.1` only; Windows Firewall does not prompt
  for loopback. Ephemeral port avoids collisions with the user's dev server.
- **`ThreadingHTTPServer` + long-lived audio range requests**: serve MP3s
  with explicit `Content-Length` and support `Range` minimally (WebView2 asks
  for ranges on `<audio>`; `http.server` doesn't do ranges natively — either
  read-whole-file responses with `Accept-Ranges: none`, which Chromium
  accepts for small MP3s, or implement single-range. Start with
  `Accept-Ranges: none`; revisit only if seeking misbehaves).

## Open questions

1. **Should the preview auto-replay narration on every live-reload tick, or
   only on explicit Replay?** Auto-replay on every debounced keystroke means
   the MP3 restarts constantly while typing. *Recommendation:* auto-play
   audio only on verse navigation and Replay; silent re-inject (animation
   restarts, audio doesn't) on edit-driven reloads. This is a one-flag
   distinction in `/state` (`cause: "edit" | "nav"`).

2. **"Edit spec" opens files with `os.startfile` — is your default `.py`
   association a real editor?** If `.py` opens the interpreter or Notepad,
   this flow is unpleasant. *Recommendation:* keep `os.startfile` as default
   but honor an optional `"editor"` command in the studio settings (e.g.
   `code -g {file}`); worth 5 lines.

3. **Re-run scope for translation-suffixed families:** the generators'
   `main()` renders *all* translations (network name-resolution per code).
   Fine for correctness, slowish for iteration. *Recommendation:* ship with
   full `main()` re-runs (zero generator changes needed); if iteration feels
   slow in practice, add a `--only-generic` flag to the two genealogy
   generators later rather than designing it in now.

4. **Marker comment format** — I chose an XML comment with a date, purely
   informational. If you ever want the *re-run* path to warn before
   clobbering marked files ("this chapter has 2 hand-edited files — re-run
   anyway?"), say so and T4 grows a pre-run scan. *Recommendation:* add the
   pre-run scan; it is cheap and it is the only guard against silently losing
   one-off edits, which is the exact failure the registry exists to prevent.

5. **Does the studio need a raster export** (SVG → WebP for the desktop
   reader) so authored visuals show on the PC app too? Out of scope for this
   design (the desktop stage renders raster only), and cairosvg is off the
   table on ARM64. If wanted later, the WebView2 preview subprocess could
   screenshot frames — a genuinely new feature, not glue. *Recommendation:*
   defer; the web app is the SVG consumer by design.
